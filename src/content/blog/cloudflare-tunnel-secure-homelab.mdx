## Why Cloudflare Tunnel?

When self-hosting services at home, the traditional approach is port forwarding on your router — open ports, point DNS to your public IP, and manage SSL certificates with Let's Encrypt. This works, but it has problems:

- Your home IP is exposed to the internet
- You need to manage certificate renewal
- ISP-level CGNAT can block incoming connections
- Every open port is an attack surface

Cloudflare Tunnel solves all of these by creating an **outbound-only connection** from your server to Cloudflare's edge network. No inbound ports needed.

## How It Works

```
Internet User → Cloudflare Edge → Cloudflare Tunnel → Your Homelab (cloudflared)
```

The `cloudflared` daemon on your server initiates a persistent connection to Cloudflare. When someone visits your domain, Cloudflare routes the traffic through this tunnel. Your server never opens a listening port to the internet.

## My Setup

I run `cloudflared` inside a dedicated Proxy LXC container in my Proxmox homelab. Here's the configuration:

### 1. Install cloudflared

```bash
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64 \
  -o /usr/local/bin/cloudflared
chmod +x /usr/local/bin/cloudflared
cloudflared tunnel login
```

### 2. Create a Tunnel

```bash
cloudflared tunnel create homelab
```

This generates a tunnel ID and credentials file. The tunnel is tied to your Cloudflare account.

### 3. Configure Ingress Rules

The `config.yml` defines which services are accessible and how traffic is routed:

```yaml
tunnel: <tunnel-id>
credentials-file: /root/.cloudflared/<tunnel-id>.json

ingress:
  - hostname: nexacampus.idev-fun.org
    service: http://web-apps:80
  - hostname: arpotret.idev-fun.org
    service: http://web-apps:80
  - hostname: grafana.idev-fun.org
    service: http://monitoring:3000
  - hostname: vaultwarden.idev-fun.org
    service: http://docker:8080
  - service: http_status:404
```

The last rule is a catch-all that returns 404 for any unmatched requests.

### 4. DNS Configuration

Point each subdomain to the tunnel:

```bash
cloudflared tunnel route dns homelab nexacampus.idev-fun.org
cloudflared tunnel route dns homelab grafana.idev-fun.org
```

This creates CNAME records pointing to `<tunnel-id>.cfargotunnel.com`.

### 5. Run as a Service

```bash
cloudflared service install
systemctl enable cloudflared
systemctl start cloudflared
```

## Security Benefits

| Traditional Port Forwarding | Cloudflare Tunnel |
|---|---|
| Home IP exposed | IP hidden behind Cloudflare |
| Manual SSL (Let's Encrypt) | Automatic TLS at the edge |
| Open ports on router | Zero inbound ports |
| Direct attack surface | Cloudflare WAF and DDoS protection |
| CGNAT can block connections | Works behind any NAT |

## Access Control with Cloudflare Access

For sensitive services like Grafana or Vaultwarden, I layer **Cloudflare Access** on top of the tunnel. This adds an authentication gate before traffic even reaches my server:

- Email OTP for admin panels
- GitHub OAuth for development tools
- IP allowlisting for internal services

This means even if someone knows the URL, they can't access the service without passing Cloudflare's auth first.

## Monitoring the Tunnel

Cloudflare provides a dashboard showing:
- Tunnel status (healthy/degraded/down)
- Request count and bandwidth
- Error rates per service
- Latency from Cloudflare edge to origin

I also run a simple health check script that pings each service endpoint and alerts me via email if any service is unreachable:

```bash
#!/bin/bash
for domain in nexacampus.idev-fun.org grafana.idev-fun.org; do
  status=$(curl -s -o /dev/null -w "%{http_code}" "https://$domain")
  if [ "$status" != "200" ] && [ "$status" != "302" ]; then
    echo "ALERT: $domain returned $status" | mail -s "Service Down" admin@example.com
  fi
done
```

## Trade-offs

Cloudflare Tunnel isn't perfect:
- **Dependency on Cloudflare** — if their network has issues, your services are down
- **Latency** — traffic routes through Cloudflare's edge, adding a small hop
- **Terms of Service** — Cloudflare's ToS restricts certain use cases (like streaming large media files through the tunnel)

For my use case — hosting web applications and development tools — these trade-offs are acceptable. The security and simplicity benefits far outweigh the downsides.
